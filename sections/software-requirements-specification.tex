\subsection{Method}

% What form of requirements gathering did you use? \label{q} \newline
In order to discover, track, and prioritize all requirements, I was asking a variety of questions --- especially in the beginning --- and kept sorting them by importance using the MoSCoW framework\cite{MoSCoWMethod2026}. Besides being very simple and clear, this method has proven itself in my previous projects, which is the main reason why I chose it as my main tool at this stage. All of this was done on Jira in a kanban board, so that it has always remained organized and visible to me and mentors.


\subsection{Requirements}

%What are the functional requirements? \label{q} \newline
%Which features/user stories did you design? \label{q} \newline
\emph{Functional requirements} can be seen in the table~\ref{tab:requirements-with-scope}. ``Must have'' requirements form the minial viable product (MVP). Out of all requirements, I have created epics that can be seen in the list below. I have not used user stories, since the requirements are very broad with many tasks.

\begin{enumerate}
    \item As a developer, \\
    I want to be able to run a federated learning simulation of apnea predictions on a single computer without additional use of docker containers, \\
    so that the whole structure is as simple as possible.
    \item As a developer, \\
    I want to be able to run a federated learning simulation of apnea predictions by using federated learning on a single device using docker/virtual machine, \\
    so that the whole project can be easily started by anyone on any machine.
    \item As a developer, \\
    I want to be able to run a federated learning simulation of apnea predictions on a single computer without additional use of docker containers, \\
    so that the whole structure is as simple as possible.
    \item As a developer, \\
    I want to have a basic security feature like differential privacy implemented on top of federated structure, \\
    so that I can assure the system also secures weights and parameters that are send to a central server.
\end{enumerate}

%What are the non‑functional requirements? \label{q} \newline
\noindent \newline
\emph{Non-functional requirements}:
\begin{itemize}
    \item A developer needs to understand
    \begin{itemize}
        \item the code,
        \item the documentation,
        \item how to interpret results at the most basic level, and
        \item how to run the project.
    \end{itemize}
\end{itemize}
